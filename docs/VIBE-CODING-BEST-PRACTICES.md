# Vibe Coding 最佳实践

> 本文以这个 SCM MCP server 项目为例，解释「为什么要有这些文件」以及「怎么做才不会翻车」。
> 适合没有系统工程背景、但想用 AI 把一个完整项目做出来的人。

---

## 一、什么是 Vibe Coding，以及它的真实风险

**Vibe coding** 是指完全依赖 AI（如 Claude、Cursor、ChatGPT）边对话边写代码，不做传统的规划和文档。感觉上很爽——直接说需求，AI 立刻产出代码。

但这个方式有一个致命缺陷：**AI 没有记忆，也没有边界感。**

没有约束的 AI 会：

| 问题 | 具体表现 |
|---|---|
| **臆造内容** | 你说"帮我调 SCM 的 API"，AI 凭印象写出根本不存在的字段名和端点 |
| **越界扩展** | 你说"加个查询功能"，AI 顺手给你加了缓存、加了鉴权、重写了一半架构 |
| **前后矛盾** | 第一次对话和第三次对话说的技术栈不一样，代码越改越乱 |
| **无法复现** | 一个月后你忘了为什么这样写，AI 也不记得，项目就此锁死 |
| **无法协作** | 别人（或另一个 AI 实例）接手时，不知道边界在哪，动什么都可能崩 |

**Vibe coding 不是问题，没有结构才是问题。**

这个项目的做法是：**给 AI 建一套「约束文件」**，让 AI 每次对话都在同一个框架里工作。

## 二、这个项目用了哪些文件，每个文件解决什么问题

```
项目根目录
├── CLAUDE.md        ← AI 的行为宪法（最重要）
├── DESIGN.md        ← AI 的工作清单（做什么）
├── WORKFLOW.md      ← AI 的操作流程（怎么做）
├── docs/
│   ├── PRD.md       ← 项目的北极星（为什么做）
│   └── BUILD-FROM-ZERO.md  ← 可复现手册（给人看的）
└── README.md        ← 给用户看的使用说明
```

下面逐一解释。

---

### 文件 1：CLAUDE.md — AI 的「行为宪法」

**解决的问题**：AI 不知道自己在做什么项目，也不知道什么不能做。

**没有它会怎样**：你说"帮我优化一下"，AI 可能把 stdio 传输改成 HTTP，把 httpx 换成 requests，顺手引入 Redis。每次都是惊喜。

**它的作用**：在每次对话开始前，给 AI 一份「不可变的约束」。

这个项目的 CLAUDE.md 包含四类内容：

```
1. 项目身份
   "这是一个 MCP server，只做 REST 转发，不是新平台"
   → 防止 AI 把它做成一个全功能后端

2. 技术栈（钉死）
   "Python + 官方 mcp SDK + stdio + httpx，不许换"
   → 防止 AI 每次选不同技术栈

3. 禁止事项（红线）
   "不臆造 schema、不硬编码地址、不加鉴权逻辑"
   → 防止 AI 越界添加功能

4. 必须执行
   "凭据走环境变量、每个 tool 必须追溯到 openapi"
   → 强制 AI 遵守关键约定
```

**写 CLAUDE.md 的原则**：写你「最怕 AI 悄悄做的事」。如果你不想让 AI 改数据库、不想让它引入新依赖、不想让它改传输协议——就写进去。

---

### 文件 2：DESIGN.md — AI 的「工作清单」

**解决的问题**：AI 不知道应该实现什么，于是自由发挥。

**没有它会怎样**：你说"实现安全策略相关的功能"，AI 可能实现 20 个 tool，也可能只实现 3 个，字段全靠猜，命名风格每次都不一样。

**它的作用**：把「要做什么」落成一张表——每一行是一个 tool，对应一个 API 端点。AI 按表实现，不多不少。

这个项目的 DESIGN.md 是这样的：

```
| MCP tool           | HTTP + 路径                | 类型  |
|--------------------|----------------------------|-------|
| list_addresses     | GET /addresses             | 只读  |
| create_address     | POST /addresses            | ⚠️ 写 |
| get_address        | GET /addresses/{id}        | 只读  |
...
```

**关键设计决策：schema 来源唯一**。这个项目的所有字段定义，都来自 `openapi-specs/scm/` 下的 YAML 文件，不允许 AI 凭感觉填字段。这条规定防止了最常见的 vibe coding 事故：AI 写出的代码在你的系统里根本跑不通，因为字段名对不上。

**写 DESIGN.md 的原则**：先把「做什么」定清楚，再让 AI 动手。这张表就是你和 AI 之间的合同。

---

### 文件 3：WORKFLOW.md — AI 的「操作流程」

**解决的问题**：AI 不知道该先做什么、后做什么，常常跳步。

**没有它会怎样**：AI 直接开始写代码，结果发现 API 文档没读，写出来的代码跑不通，再回头改，越改越乱。

**它的作用**：定义固定的三阶段顺序——

```
阶段 1（契约同步）：先读 API 文档，把 DESIGN.md 填完整
       ↓
阶段 2（实现）：按 DESIGN.md 写代码
       ↓
阶段 3（验收）：对照验收清单逐项检查
```

这个顺序**不可颠倒**。如果你让 AI 跳过阶段 1 直接实现，它就会开始臆造字段。

WORKFLOW.md 还有第二个作用：**记录验收结果**。每个阶段完成后，把验收结论填进去，形成项目的质量记录。这样下次接手的人（或另一个 AI 实例）能看到「这个东西做到什么程度、验证过什么」。

**写 WORKFLOW.md 的原则**：写你希望 AI 每次遵守的操作步骤，以及每步完成的标准是什么（不能是"感觉差不多了"，要是可以执行的判定）。

---

### 文件 4：PRD.md — 项目的「北极星」

**解决的问题**：做着做着不知道为什么在做这件事，功能越加越多，边界越来越模糊。

**没有它会怎样**：你让 AI "再加一个功能"，它加了。再加一个，它也加了。三个月后这个项目变成了一个没人能维护的大泥球，而且没人说得清它到底是给谁用的。

**它的作用**：PRD（产品需求文档）在 vibe coding 里的作用不是给产品经理看的，而是**给 AI 划定边界**。

这个项目的 PRD.md 包含：

- **目标用户**：网络安全工程师、平台运维、安全架构师、IAM 管理员——而不是「所有人」
- **In Scope**：明确列出 MVP 必须有的能力（C1-C5）
- **Out of Scope**：明确写出不做什么（SASE 部署、跨租户合并、自动分页……）以及**为什么不做**
- **验收标准**：7 条，每条都是可执行的二元判定，不含"好用""流畅"这类主观词

**Out of Scope 比 In Scope 更重要**。人们总是容易往项目里加东西，从不容易做减法。把「不做」写进文件，是防止功能蔓延（feature creep）最有效的手段。

**写 PRD 的原则**：把你最常听到的"顺便也做一下……"全部写进 Out of Scope。

---

### 文件 5：BUILD-FROM-ZERO.md — 可复现手册

**解决的问题**：这个项目是怎么做出来的，别人能不能复现，教训有没有沉淀。

**它的作用**：记录每个步骤的**真实提示词**（逐字，可直接复制）、教学点和验收命令，再加上每步的 `git commit` / `git tag` / `git push`。

为什么要保存真实提示词？因为**提示词是这个项目最有价值的资产之一**。代码会过时，API 会变，但「如何和 AI 合作完成这类项目」的方法论是可以复用的。把提示词存下来，下次做类似项目时直接参考，不用从零摸索。

为什么要打 tag？因为**可以随时回到任意历史状态**。`git checkout step-3` 就能看到"只有文档骨架、还没写任何 tool"的状态，用于对比、演示或教学。

---

### 文件 6：README.md — 给用户的使用说明

**解决的问题**：项目做完了，别人不知道怎么用。

**它的作用**：安装步骤、环境变量说明、注册到 MCP 客户端的配置、可用 tool 列表。这个文件是写给「用这个工具的人」看的，不是给开发者看的。

## 三、这套文件系统背后的核心逻辑

可以用一个比喻来理解：**雇一个很聪明但完全不了解你业务的新员工。**

| 新员工需要什么 | 对应文件 |
|---|---|
| 公司规章、不能做什么 | CLAUDE.md |
| 这次任务的工作清单 | DESIGN.md |
| 做事的标准流程 | WORKFLOW.md |
| 这个产品是给谁做的、边界在哪 | PRD.md |
| 上一个人怎么做的，踩过什么坑 | BUILD-FROM-ZERO.md |
| 用户手册 | README.md |

AI 每次对话都是「新员工第一天」——它不记得上次说了什么。这些文件就是让 AI 每次都能快速进入状态、在正确的轨道上工作的「入职材料」。

## 四、三个最常见的 Vibe Coding 事故，以及这套系统如何预防

### 事故一：AI 写出根本跑不通的代码

**原因**：AI 没有读 API 文档，靠训练数据里的"印象"写字段名，字段名对不上真实 API。

**预防**：CLAUDE.md 红线「不臆造 schema，唯一来源是 openapi YAML」+ WORKFLOW.md 阶段 1「先读文档再实现」。

---

### 事故二：项目越改越大，最后没人能维护

**原因**：没有边界，AI 每次都"顺便"加功能，代码复杂度指数增长。

**预防**：PRD.md 的 Out of Scope 列表 + CLAUDE.md 红线「只做 REST 转发，不重写业务逻辑」。

---

### 事故三：做到一半，前后不一致

**原因**：多次对话之间 AI 忘记了之前的决策，技术选型漂移，代码风格割裂。

**预防**：CLAUDE.md 钉死技术栈 + DESIGN.md 统一命名规范 + git tag 保存每个阶段的快照。

## 五、实际操作：怎么把这套方法用在自己的项目里

### 第一步：先写约束，再动代码

在写任何代码前，先花 30 分钟完成这三件事：

1. **写 CLAUDE.md**：填写项目身份（这是什么、不是什么）、钉死技术栈、列出红线。
2. **写 PRD.md**：目标用户是谁、In Scope 有什么、Out of Scope 排除什么。
3. **写 DESIGN.md 骨架**：列出要实现的功能清单，标注数据来源（不臆造）。

这三个文件加起来不到一小时。但它们能救你几十小时的返工时间。

### 第二步：让 AI 先给计划，你确认后再执行

不要直接说"帮我实现 X"。要说：

> "根据 CLAUDE.md 和 DESIGN.md，先给我实现计划（分几步，每步做什么），**等我确认后再执行**。"

这个「等我确认」的 gating，是防止 AI 跑偏最有效的单个操作。一旦 AI 开始执行，中途叫停的成本很高；但在计划阶段修正，只需改几行文字。

### 第三步：每完成一个阶段，立刻提交

```bash
git add -A
git commit -m "step-N: 描述"
git tag step-N
git push
```

这不只是版本控制的好习惯——它强制你在继续之前确认「这一步确实做完了」。没有 tag 的 vibe coding 项目，往往在某个节点陷入混乱后，连退到哪个版本都不知道。

### 第四步：验收标准必须是「可执行的二元判定」

不要写"功能正常"，要写：

> "调用 `list_addresses`，传入 `folder='Prisma Access'`，返回 HTTP 200，响应体包含 `data` 数组。"

模糊的验收标准让 AI 自我评估时永远是「通过」。具体的判定条件才能发现真实问题。

## 六、一句话总结

> **Vibe coding 的本质是和 AI 协作，协作的前提是双方在同一个框架里工作。这些文件就是那个框架。**

没有框架的 vibe coding 是「和一个失忆的天才对话」——每次都要重新解释背景，每次都可能得到不同的答案，项目质量完全取决于运气。

有了框架的 vibe coding，是「带着一个聪明的新员工，按流程把事做完」——AI 的能力被约束在正确的方向上，你的时间花在决策上，而不是反复纠错上。

## 附：这个项目的文件层级关系

```
CLAUDE.md（L1）    ← 最高约束，其他文件不得违反
    │
    ├── DESIGN.md（L2）   ← 服从 L1，定义「做什么」
    │       └── 每个 tool → 一个 API 端点（来自 openapi YAML）
    │
    ├── WORKFLOW.md（L3）  ← 服从 L1/L2，定义「怎么做」
    │       └── 阶段 1 → 2 → 3，顺序不可颠倒
    │
    └── docs/PRD.md        ← 服从 L1，定义「为什么做、边界在哪」
            └── Out of Scope 比 In Scope 更重要
```

当这些文件之间发生冲突时，**上层优先**：L1 > L2 > L3。这个规则让 AI 知道「如果有疑问，看哪个文件」，而不是在矛盾中自由发挥。
